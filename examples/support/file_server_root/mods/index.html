Modding and Customization

Real-world applications have specific needs far exceeding simple file/application servers.

Request-Response Cycle Event Hooks

One of the primary goals of the Aerys project is to make application-specific customization as simple as possible. To accommodate this goal the following events are broadcast to afford userland code the chance to modify the Aerys server's behavior:

  • onHeaders
  • beforeResponse
  • afterResponse

Mods registered for the relevant HTTP request's host are invoked at each step allowing application developers to tailor solutions for project-specific needs.

Built-in Mods

As an example of what's possible on the Mod front (and because they're just really useful) Aerys comes prepackaged with the following Mods:

ModEvent(s)Description
ErrorPagesbeforeResponse Intercept error responses before they're sent to the client and assign user-defined entity bodies for error respones (400-599).
ExpectonHeaders Specify callbacks to evaluate request headers and return a boolean on a per-resource basis to answer 100-continue requests without cluttering your application code.
LimitonHeaders Customizable rate-limiting by IP with support for multiple interval/rate combinations. Rate-limited clients receive a standard 429 Too Many Requests response when exceeding defined rate thresholds.
LogafterResponse Customizable per-request logging to multiple files and/or processes with built-in support for standard logging formats such as the NCSA Common Log Format and the NCSA Combined Log Format.
SendFilebeforeResponse Significantly speeds up file-serving when the static server isn't an option due to login/validation/other application constraints. Application handlers can simply specify a path to the file they wish to return to the client in the X-Sendfile header and Aerys will take care of the rest. The SendFile Mod also provides all the benefits of the built-in static file server (caching headers, byte-range requests, conditional retrieval, etc).
WebsocketonHeaders While you'll often specify a Websocket handler on its own named virtual host, the Websocket Mod allows applications to overlay websockets endpoint resources on the same host as another application handler (e.g. a static file or application server). For example, you might serve static files from http://mysite.com/ but overlay a websocket chat endpoint at http://mysite.com/chat using the Websocket Mod.

Registering Custom Mods

Aerys doesn't limit applications to built-in mods. Adhering to simple interfaces means you can implement and deploy your own mods. The following interfaces are used to define Mod functionality:

  • Aerys\Mods\OnHeadersMod
  • Aerys\Mods\BeforeResponseMod
  • Aerys\Mods\AfterResponseMod

Custom Mod Example

Let's look at a simple mod example. The below class, when registered, will examine every response before it is output to the client. If the response assigned for the givent request ID has a status code equal to 200, the mod replaces the body with some HTML of its own. 

<?php
use Aerys\Server, Aerys\Mods\BeforeResponseMod;

class MyZanzibarMod implements BeforeResponseMod {
    
    private $server;
    private $beforeResponsePriority = 50;
    
    function __construct(Server $server) {
        $this->server = $server;
    }
    
    function beforeResponse($requestId) {
        $asgiResponse = $this->server->getResponse($requestId);
        list($status, $reason, $headers, $body) = $asgiResponse;
        if ($status == 200) {
            $newBody = '<html><body><h1>ZANZIBAR!</h1></body><html>';
            $headers['CONTENT-LENGTH'] = strlen($newBody);
            $headers['CONTENT-TYPE'] = 'text/html; charset=utf-8';
            $newAsgiResponse = [$status, $reason, $headers, $newBody];
            $this->server->setResponse($requestId, $newAsgiResponse);
        }
    }
    
    function getBeforeResponsePriority() {
        return $this->beforeResponsePriority;
    }
}

While this example is a bit spurious, it demonstrates what's possible using mod hooks. These events make it trivial to alter, analyze, log or cache any requests or responses that run through the Aerys server.
